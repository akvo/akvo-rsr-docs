  • Akvo RSR
    • Features
      • Visualising Development Projects
        • Project Pages
          • Project pages are public pages on the internet
          • Projects contain information about individual projects.
          • Projects can be planned, being executed or completed.
          • Projects can contain a mix of text and image information.
          • Projects can be followed by use of a unique RSS feed.
        • Funding Pages
          • Funding pages are pulic pages on the internet.
          • Funding pages show where funds to sustain a project are/were sourced.
          • Funding pages show where the projects funds are intended to be spent.
          • Funding pages can contain links to Governmental funding information via IATI Identifiers
        • Organisation Pages
          • Organisation pages are public pages on the internet.
          • Organisation pages show the details of organisations working on one or more Akvo RSR Projects.

      • Publishing Field Updates
        • Creation
          • Authentication
            • User Registration
              • Users must register before they are able to create updates
              • Users must belong to an organisation linked to the project to be able to update
            • Login
              • Users must have successfully logged in before creating updates
              • Users can login on the Project Page or via the Admin interface
              • The Login details will be applied to the Project Update record
          • Text Formatting
            • Text entered into Projects and Updates should have a limited formatting set of options
            • There should be a WYSIWYG editor for formatting text
        • Editing
          • Limits
            • Updates can only be edited for the first 30 minutes after creation
            • Project Title is limited to 50 characters
            • Photo Caption is limited to 75 characters
            • Photo Credit is limited to 25 charcters
            • Update Text is not limited
          • Version Control
            • We currently store no version control on updates.
            • Updates should be recorded for each modification that is made.
            • Each version should be stored.
            • Previous versions should be visible to users with the permissions to see the update at the time the update is made.
        • Media
          • Pictures
            • Format
              • Photo should be in 4:3 ratio
              • Photo needs to be in jpeg, gid or png
            • File Size
              • There should be a reasonable size limit on the photos (2MB)
              • Thumbnails and other available sizes are created on file upload
              • All size versions are stored
          • Videos
            • Compatibility
              • All videos should be hosted on a 3rd Party Video Hosting site
              • RSR Supports YouTube, Vimeo & Blip video embedding
              • All URL possibilities from these hosting providers should be supported
            • Limits
              • Videos must be publicly viewable

      • Fund Raising
        • It is possible for Partners to utilise Akvo RSR to raise funds for their projects
        • RSR has a donations module allowing for funds to be donated to projects
        • Donations can be made using PayPal or iDeal
        • Transaction charges are at the expense of the donor
        • Donors can choose to remain annonymous
        • Organisations can also donate directly to project via a Bank Transfer which is currently a manual process
        • Organisations can be publicised as a funding partner on projects they have contributed to
    • Functionality
      • User Registration
        • Requirements
          • New users need to be able to register on Akvo RSR
          • Users must be assigned to a single Organisation
          • Users are given a set of permissions based on pre-configured roles
          • Users must have a valid email address
          • New Users must be identified and validated by an Akvo or Organisation Administrator
        • Process
          • New Users should first select an Orgnaisation they would like to register about
          • Users then enter their personal information
            • Username
            • Password
            • First Name
            • Last Name
            • Email Address
          • A request is submitted for the account to be validated
          • An Organisation Administrator should validate the account, confirming they agree to the user having access
          • An Akvo Administrator can validate the account in the event that there is no relevant Organisation Administrator to perform this task
          • The User receives an email informing them that their account has been validated
      • User Login
        • Requirements
          • Users must be able to log in to Akvo RSR ftrom any page
          • The same login should be used for both the Site and the Admin sections
          • There should be a process to reset a users' password
          • The login process should be secure
        • Process
          • Visitors to the site should be presented with a login option on each page
          • The Login option could be a link to a login page
          • The Login option could be 2 fields and a submit button for Email/Username and Password
          • If the correct details are submitted then the user should be logged into the system
          • If the incorrect details are submitted the user should be requested to re-enter the details
          • The user should also be given the option to reset their password at this stage
      • Organisation Registration
        • Requirements
          • Organisations should be real and existing organisations
          • Organisations should have a valid Geolocation
          • Organisations can upload a logo file to be displayed
          • Details of a Contact Person are required
          • An unlimited Description field can be completed
        • Process
          • Organisations can be linked directly to Akvo
            • Organisation agrees to use Akvo RSR
            • Organisation sends information to Akvo Admin including Contact Information and Logo
            • Organisation is created in Akvo RSR
            • The first user is asked to sign up via Akvo.org to the Organisation
            • User is approved by Akvo Admin and if appropriate is allocated the role of Organsiation Admin
          • Organisations can also be linked via an existing RSR Partner
            • Organisation is included within Project Plans of an Akvo RSR Partner
              • Usually a Support Partner
            • Organisation details are received from Support Partner
            • Akvo Admin create Organisation in RSR
            • Users are encouraged to sign up at Akvo.org within the new Organisation
            • New users are approved by Akvo Admin with confirmation acquired from Support Partner Admin if necessary
      • Project Registration
        • Requirements
          • All Projects must have a Title
          • New Projects have the following Required Fields
            • Title (45)
            • Subtitle (75)
            • Category
              • Categories need to be selected from the existing list of Categroeis within RSR
            • Summary of Project Plan (400)
            • Sustainability
            • Overview of Goals (600)
            • Partnership Information
              • This is the Partner Type of the Creating organsiation
          • Benchmarks can only be completed once a Project has been saved and linked to its categories
        • Process
          • A new project is created here: http://akvo.org/rsr/admin/rsr/project/add/
          • The project is given a Title of 45 characters or less
          • The project is given a Subtitle of 75 characters or less
          • The project is given a Status
            • Needs Funding
            • Active
            • Completed
            • Cancelled
            • Archived
          • The project is allocated to one or more RSR Categories
            • Categories are linked to Focus Areas
            • Akvo RSR currently has 5 Focus Areas
              • Water & Sanitation
              • Education
              • Economic Development
              • Healthcare
              • IT and Communication
            • Categories determine which indicators/benchmarks are available
          • The project is given a Summary of Project Plan of 400 characters or less
          • The user uploads an image file for the project
          • The image is given a caption
            • The caption is currently not published anywhere and so is not necessary to be included
          • The project is given a Background of 1000 characters or less
          • The project is given a Project Plan
          • The project is given a Current Status of 600 characters or less
          • The project is given a Sustainability
          • The project has 2 meta info options which are not published
            • Project Rating
              • The project rating is a number
              • The project rating is currently not used
            • Notes
              • Notes can be used by Admin and other RSR Admin users to comment on projects
              • Notes are not published and are only visible via the Admin
          • The project is allocated a currency
            • Projects can be in Euro €
            • Projects can be in Dollars $
          • The project is given a Date Request Posted
            • This field is used in different ways by different users
              • This field is sometimes used as the starting date of the project
              • This field is sometimes used as the posting date of the project
              • This field is sometimes used as the date the project was funded
            • This field should be standardised to have a single purpose
          • The project is given a Date Complete
            • This should be the expected completion date
            • This should be updated to the actual closing date when the project is completed
          • The project is given an Overview of Goals of 600 characters or less
          • The project is given up to 8 goals of 100 characters or less each
          • The project is given Budget Items
            • Each budget item consists of a Label and an amount
            • There is an Other Label which allows for Customised Budget items
          • The project is given Links
            • Links can be to internal Akvopedia pages
            • Links can be to external websites
            • Links should direct users to additional relevant information
              • This could be about the project
              • This could be about the participating organisations
          • The project is given all additional project partners
            • Each partner is allocated a Type
              • Field Partner
              • Funding Partner
              • Sponsor Partner
              • Support Partner
            • Funding Partners can be allocated an amount that they have contributed ot the project
            • Field Partners can be given a Partner Type Extra
              • Alliance
              • Knowledge
              • Network
            • IATI Activity IDs can be added per partner
              • This is the ID the Partner has allocated or been allocated to the project within the IATI Registry
            • Internal IDs can be added per partner
              • This is any identifier the Partner might use to identify this project
            • An IATI URL can be added where the Published IATI information from this partner for this project can be found
          • The project is given Locations
            • If a project has a location, then one and only one location must be a Primary Location
            • Locations consist of multiple parts
              • Coordinates
                • Latitude
                • Longitude
              • Address
                • City
                • State
                • Country
                • Address 1
                • Address 2
                • Postcode
            • Coordinate information is used by RSR to put pins onto maps
            • The Address information is used to display on the project page and for filtering of data
            • Additional locations can be added to projects
          • The project must then be saved
          • The project can then have Benchmarks populated
            • Each benchmark consists of an Indicator
              • Indicators are linked to Categories
            • Each Benchmark also has a field for a number to be entered
              • The number indicates the target to be reached for the project
      • Project Updates
        • Requirements
          • Updates can only be added to Published Projects
          • Updates can only be added by authorised RSR users
          • Update users must belong to one of the Partners linked to a project
          • Updates can be entered via the Web Interface or via the Admin
          • Updates can be modified for the first 30 minutes after publishing via the Web Interface
          • Updates can be edited afterwards via the RSR Admin for those users with sufficient permissions
          • Updates support Markdown Formatting
            • We currently have no editor for Markdown Formatting
            • Formatting must be known to the user
            • We have need of a WYSIWYG editor for Update entry
          • One image can be added per Update
          • Videos can be embedded from a selection of Hosting providers
            • YouTube
            • Vimeo
            • Blip
          • Media can be placed above or below the content of the Update
        • Process
          • User navigates to the Project to be updated in RSR
          • User clicks to Add an update to the project
          • The User is prompted to login if not already
          • RSR checks to ensure that the user has the required permissions to update the project
            • If not, then RSR presents the user with the Forbidden page (403) page
          • The user then enters the Update information
            • Title (50)
            • Text
            • Photo
            • Photo Caption (75)
            • Photo Credit (25)
            • Video URL
            • Video Caption (75)
            • Video Credit (25)
          • The user selects whether the media should be placed at the top or the bottom of the update
          • The update is saved
          • The user is informed that they have 30 minutes to edit the update in case anything needs to be changed
      • Maps
        • Requirements
          • Displaying object location coordinates
          • Zooming to an appropriate level for the pins and area displayed on the map
          • Displaying multiple pins per object when appropriate
          • Zooming available to users
          • Navigation of the map available to users
        • Process
          • User loads a page within RSR that contains a map
          • The map sources the data from the RSR Database
            • This is currently obtained from the RSR API
          • The map displays the pins in the correct Geolocations for the user to see
          • The map is styled as per the settings and Google Maps default

      • Donations
        • Requirements
          • Donations to individual projects need to be made via the Website
          • Donors should not have to register on Akvo.org to donate
          • Donors should have the option of donating via PayPal or iDeal
          • Donors should have the option of donating annonymously
          • The donation process should be secure
          • Donors should be returned to their starting page after the donation process has been completed
          • Donors should be able to donate up to the total value of the project
          • Donors should be able to compelte project funding via their donations
          • Donation details should only be visible to Authorised Akvo Administrators
          • Donors should receive confirmations once donations have been completed
        • Process
          • Users select a Project they would like to Donate to
          • Users click on the Donate button on the right hand side of the project
          • Users choose which method they would like to donate using
            • PayPal
              • User enters the amount of donation up to the amount still required by the project
              • User completes their name and email information
              • User selects if they would like to remain annonymous or not
              • User is directed to PayPal to complete their payment
              • Once the process is complete, the user is directed to the Project page
              • The user receives an email from RSR once the payment has been confirmed and the donation is accepted
              • If the process is unsucessful, the donation is cancelled and the user receives a failure notification
            • iDeal
              • User enters the amount of the donation up to the amount still required by the project
              • User completes their name and email information
              • User selects if they would like to remain annonymous or not
              • User is directed to iDeal to complete their payment
              • Once the process is complete, the user is directed to the Project page
              • The user receives an email from RSR once the payment has been confirmed and the donation is accepted
              • If the process is unsuccessful, the donation is cancelled and the user receives a failure notification
            • Becoming a Funding Partner
              • User downloads the Akvo Donation Form available here: http://www.akvo.org/web/sites/default/files/Akvo_donation.pdf
              • User fills in the form using Adobe Acrobat Reader
              • User transfers the total donation amount to the bank account mentioned in the form
              • Once the donation is confirmed the Donor is setup as a Funder on the Project
              • Information on how to embed projects into websites is provided to the user
              • The Project Partnners are notified and the funds are transferred from Akvo to the Receiving party on the Project

      • API
        • Requirements
          • Read
            • Information within the Akvo RSR DB should be accessible via an API
            • Project information in particular is important to be accessible
            • Projects should be grouped into a data set with all linked records and data allowing a single call for all related project information
            • No user information should be accessible without a valid API key
            • API Keys should only permit access to secure data for the owner organisation
            • The API should support JSON and XML Calls
          • Write
            • It should be possible to create new DB objects via the API
            • New objects must be created by an authorised API user
            • Projects should have all attributed data included within the call to create
            • The API should accept directly or indireectly the IATI XML File Format

          • RSS Feeds should contain complete information about DVB Objects
          • RSS Feeds should contain links and images included within the data

      • Administration
        • Requirements
          • Permissions
            • Administration of RSR should be possible by both Organisation and Akvo Administrators
            • The set of functions available to Organisation Administrators should be limited
            • Administration permissions should be divisible into Roles
            • Roles should be allocated to Admin users
            • Individual Permissions should be assignable to roles
            • An additional Role of Superuser should be available for more complex operations
            • No standard user should have the Superuser role
          • Users
            • User Administration within Organisations should be possible for Organisation Administrators
            • Akvo Administrators should be able to perform user Administration for all users
            • Users information should not be changed without the users permission
            • New users should not be entered using the Admin
            • New users need to be confirmed via the Admin
            • New user notifications should be deliveed automatically upon confirmation by the system
              • Not implemented
          • Counters
            • Counters for projects and other objects should be visible via the Admin
            • View Counters should show the number of visitors to the object
            • View counters should not count visitors using the same Mac Address
              • Other identifiers could be used to identify individual users
          • RSR
            • RSR Content should all be Administered and Configurable via the Admin
            • All Projects and User entered data should be accessible to Organsiation Admin
            • All fixed data, such as Benchmarks, Focus Areas etc, should only be accessible to Akvo Admin
            • Each record should be editable and saveable
            • New records should be addable for user generated content
        • Process
          • User logs into RSR Admin at http://akvo.org/rsr/admin
          • User is presented with the Admin homepage indicating all of the sections they have access to
          • Each section allows Admin of an Object Type or set
          • It is possible to add new objects but this should only be performed for limited objects
            • All Admin
              • Partner Sites
              • Projects
              • Updates
            • Akvo Admin
              • Benchmark Names
              • Budget Item Labels
              • Categories
              • Countries
              • Focus Areas
              • Organisations
